  1 ABSTRACT
  2 
  3     WindowMasker is a program that identifies and masks out highly 
  4     repetitive DNA sequences and DNA sequences with low complexity 
  5     in a genome using only the sequence of the genome itself.
  6     WindowMasker is described in [1]. Please cite this paper in
  7     any publication that uses WindowMasker.
  8 
  9     The statically linked binary version of windowmasker program
 10     compiled for Linux platform is available via FTP at
 11 
 12     ftp://ftp.ncbi.nlm.nih.gov/pub/agarwala/windowmasker/windowmasker
 13 
 14 
 15 [1] Morgulis A, Gertz EM, Schaffer AA, Agarwala R. WindowMasker:
 16     Window based masker for sequence genomes. Submitted for publication.

 25

 28 DESCRIPTION
 29 
 30     WindowMasker has two modules for masking DNA sequences.  The 
 31     WinMask module is used to mask potentially repetitive sequences 
 32     by counting the number of times different n-mers (units) occur 
 33     in the genome. The DUST module is used to identify and mask low-
 34     complexity regions. 
 35     
 36     The WinMask module works in two stages. During Stage 1, unit 
 37     counts are collected and stored in a separate file. During 
 38     Stage 2 that file is used to mask the input sequences. Usually 
 39     the unit counts file is created once per genome and then used
 40     multiple times for masking. In addition, an option is provided
 41     to convert between different unit counts formats.
 42 
 43     Stage 1 is selected with "-mk_counts" flag. Stage 1 processes
 44     input data in up to 4 passes.
 45 
 46         Pass 1 (optional). Checking for long possibly duplicate 
 47         sequences in the input. This pass is made when 
 48         "-checkdup true" is selected.
 49 
 50         Pass 2 (optional). Compute the total size in bases (bp) of 
 51         the genome.  This pass is made when neither unit length 
 52         ("-unit") nor the genome size ("-genome_size") are given on 
 53         the command line. Knowledge of genome size is necessary to 
 54         compute the unit length automatically.
 55 
 56         Pass 3 (optional). Compute unit score thresholds T_threshold, 
 57         T_extend, T_low, and T_high. In particular the values of 
 58         T_low and T_high are necessary for the final pass to save only 
 59         the counts for the units with score above or equal to T_low 
 60         and to change the counts of units that are below or equal to 
 61         T_high to the value of T_high. This pass can be avoided by 
 62         using "-t_low" and "-t_high" command line options that 
 63         explicitly set the values of T_low and T_high.
 64 
 65         Pass 4. Generate the file containing the unit length, 
 66         significant (above T_low) unit counts, and threshold values.
 67         The counts of units that appear more than T_high times
 68         in the genome are assigned the count value of T_high.
 69         
 70     Stage 2 is enabled by providing "-ustat <stat_file>" argument.
 71     WindowMasker reads the data generated in Stage 1 and a set of 
 72     input DNA sequences to output information about masked 
 73     subintervals. If "-dust true" is specified, then 
 74     the corresponding algorithm of the DUST module is applied to the 
 75     input sequences in addition to window based masking. When DUST 
 76     module is run, the results of the DUST and WinMask modules are 
 77     merged together in the output. Specifically, a base is masked if 
 78     it is masked by either DUST or by WinMask.
 79 
 80     The unit counts format conversion function is triggered by
 81     "-convert" flag.

 83 OPTIONS
 84 
 85     In this section, command line options to WindowMasker are 
 86     explained. There are two subsections for Stage 1 options and 
 87     Stage 2 options. Some options are applicable in both stages 
 88     but could have somewhat different meaning.
 89 
 90 
 91 Options Selecting WindowMasker Mode of Operation
 92 
 93     -mk_counts 
 94 
 95         The presence of this flag enables Stage 1 processing
 96         (counts creation).
 97 
 98     -convert
 99 
100         If this flag is given, windowmasker functions as a converter
101         between different supported counts formats. In this case
102         options other than "-in", "-out", "-sformat", and "-smem" are 
103         not supported.
104 
105     -ustat unit_counts
106 
107         This option enables Stage 2 processing. It specifies
108         the name of the file containing the unit counts for the
109         genome in use. Unit counts are typically created during
110         Stage 1 of WindowMasker processing. The format of the unit
111         counts file is recognized automatically
112 
113 Common Options
114 
115     -exclude_ids exclude_id_list
116 
117         default: <empty>
118 
119         The name of the file containing the list of sequence ids (one 
120         per line) in the input that should not be processed.  See 
121         section FILES for an example. This option and "-ids" option 
122         are mutually exclusive.
123 
124     -ids id_list
125 
126         default: <empty>
127 
128         The name of the file containing the list of sequence ids (one 
129         per line) in the input that should be processed. See section 
130         FILES for an example. This option and "-exclude_ids" option 
131         are mutually exclusive.
132 
133     -use_ba true/false
134 
135         default: true
136 
137         Optimize unit counts data structure performance using bit arrays
138         if possible. This increases the masking speed by about 20% but
139         also increases the process memory footprint. This optimization
140         is currently available only for unit counts database created
141         with "-sformat obinary" option.
142 
143 Stage 1 Options
144 
145     -checkdup true/false
146 
147         default: false
148 
149         If the value is true, then a pass is made to check the input 
150         for long possibly duplicate sequences. The algorithm is 
151         heuristic. For a pair of sequences a match is reported if at 
152         least 5 100 bp segments approximately 10000 bp apart match 
153         exactly. "Approximately" above means that some fuzziness is 
154         allowed in the distance between consecutive 100 bp segments. 
155         This criterion will detect all duplicates as well as some 
156         false positives. In our experience, duplicates arise when 
157         multiple, distinct assemblies of some part (e.g., one 
158         chromosome) of the genome pieces are considered as part of 
159         the sequenced genome.
160 
161     -fa_list true/false
162 
163         default: false
164 
165         If the value of this parameter is true, then "-in" option
166         specifies a file containing a list of pathnames (one pathname
167         per line) to the FASTA formatted files that will be used as 
168         input. Otherwise "-in" option specifies a single FASTA
169         formatted file.
170 
171     -genome_size genome_size
172 
173         default: <empty>
174 
175         Explicitly specify the genome length in number of bases.  If 
176         this parameter is not used then the genome length is computed 
177         as the sum of lengths of all input sequences.  This parameter 
178         does not have any effect if "-unit" option is specified. 
179         Otherwise it is used to compute the unit length.
180 
181     -in input_file_name
182 
183         default: <empty>
184 
185         The location of the input file. The contents of the input 
186         file depends on the value of "-fa_list" and "-convert" options. 
187         
188         If "-fa_list false" is specified, input file is a single FASTA '
189         formatted file. Unit counts will be generated based on the 
190         sequences in that file.  If "-fa_list true" is specified, then 
191         input file contains a list of pathnames (one per line) of FASTA 
192         formatted files that form the genome. In this case unit 
193         counts will be generated based on all the sequences contained 
194         in the listed files.
195 
196         If "-convert" is specified, then "-in" option is required
197         and its value must be the name of a valid unit counts file.
198 
199     -infmt input_format
200 
201         default: fasta
202 
203         The possible values are "fasta" for reading sequence data from
204         FASTA formatted file, and "blastdb" for reading sequence data
205         from a BLAST database. 
206 
207     -mem available_memory
208 
209         default: 1536
210 
211         Assume that this much RAM is available for WindowMasker.  The 
212         value is in megabytes. Depending on the amount of available 
213         memory, passes 3 and 4 of stage 2 could contain additional 
214         subpasses.  This is especially true for large (>=14) values 
215         of unit length.
216 
217     -out output_file_name
218 
219         default: <stdout>
220 
221         The location of the output file. This parameter is optional
222         and standard output is used if it is not specified.  Section 
223         FILES contains the description of format of the file.
224 
225         If "-convert" option is given then "-out" option is
226         required and specifies the name of the target output unit
227         counts file.
228 
229     -sformat unit_counts_format
230 
231         default: ascii
232 
233         The format in which the unit counts data should be generated.
234         The possible values are "ascii", "binary", "oascii", or 
235         "obinary". "ascii" makes windowmasker generate unit counts 
236         in human readable text form, which is highly portable but 
237         slow to load during Stage 2.  The "binary" format is not 
238         portable but loads very fast. The last two formats 
239         correspond to optimized (via hash tables) unit counts data
240         structures. They require more memory to create during Stage 1.
241         However using them results in 2.5 - 4 times imrovement in
242         performance of the WinMask module. "obinary" stores data in 
243         a binary form which is not portable. "oascii" stores data 
244         in a portable text format at the expense of startup performance
245         during Stage 2.
246 
247         If "-convert" option is specified, then "-sformat" 
248         determines the conversion target format.
249 
250     -smem available_memory
251 
252         default: 512
253 
254         This option is ignored for "ascii" and "binary" unit counts
255         formats. For "oascii" and "obinary" this option specifies
256         the upper limit (in megabytes) for the size of the unit counts 
257         data structure in memory. WindowMasker will try to produce 
258         the data structure smaller than the specified size. If that
259         is not possible, error is reported.
260 
261     -t_low T_low
262 
263         default: <empty>
264         
265         Save only information about units with counts equal or bigger
266         than T_low. If "-t_low" is not specified on command line, then
267         its value is computed so that 90% of all units present in the 
268         genome have counts below T_low.
269 
270     -t_high T_high
271 
272         default: <empty>
273 
274         The units that appear more than T_high times in the genome
275         are given the count value of T_high. If "-t_high" is not 
276         specified on command line, then its value is computed so that
277         99.8% of all units present in the genome have counts below
278         T_high.
279 
280     -unit unit_length
281 
282         default: <empty>
283 
284         Specifies the unit length to use. The value is an integer 
285         between 1 and 16. If this parameter is not specified then
286         unit length is computed automatically from the genome length 
287         L in bases as the smallest number N such that L/4^N > 5.
288 
289 Stage 2 Options
290 
291     -dust true/false
292 
293         default: false
294 
295         Use the symmetric algorithm of the DUST module in addition 
296         to the WinMask module to mask out low-complexity regions. 
297 
298     -in input_file_name
299 
300         default: <stdin>
301 
302         Specifies the input file for masking. WindowMasker accepts 
303         input in FASTA format. If this option is skipped, then the 
304         standard input will be used as the source of input. 
305 
306     -infmt input_format
307 
308         default: fasta
309 
310         The possible values are "fasta" for reading sequence data from
311         FASTA formatted file, and "blastdb" for reading sequence data
312         from a BLAST database. 
313 
314     -outfmt output_format
315 
316         default: interval
317 
318         The possible values are "interval" and "fasta". If "fasta" 
319         is selected the output is a FASTA formatted file with masked 
320         regions in lower case letters. If the interval format is 
321         selected the output is a file containing, for each processed 
322         sequence, a sorted set of continuous masked subsequences.  
323         For the format of the intervals file see section FILES.  
324         [NOTE: Case of the input does not matter. All input sequences 
325         are considered unmasked and window masker does not preserve 
326         the case of the input.]
327 
328     -out output_file_name
329 
330         default: <stdout>
331 
332         Specifies the output file. The output file format depends on
333         the value of "-outfmt" option. If this option is skipped,
334         then the standard output will be used as the output file
335         descriptor.
336 
337     -set_t_high score
338 
339         default: T_high
340 
341         The score value for units with unit count above T_high.
342         See [1] for details.
343 
344     -set_t_low score
345 
346         default: (T_low + 1)/2
347 
348         The score value for units with unit count below T_low.
349         See [1] for details.
350 
351     -t_extend T_extend
352 
353         default: <the value from unit counts file>
354 
355         This parameter defines whether the interval between two
356         windows with score higher than T_threshold is masked. The
357         interval will be masked if every base of the interval lies 
358         within a window of score greater or equal than T_extend.  For 
359         the definition of the window score see description of 
360         "-window_size" option. This parameter overrides the value
361         contained in the unit counts file.
362 
363     -t_high T_high
364 
365         default: <the value from unit counts file>
366 
367         This parameter overrides the value contained in the unit
368         counts file. Since the unit counts file does not contain
369         unit counts values larger than T_high value computed
370         at Stage 1, it only makes sense to specify a T_high value
371         that is less than the one in the unit counts file. Units 
372         with counts greater than T_high will not be read from the 
373         unit_counts file and the value of "-set_t_high" will be used 
374         as the result of unit count lookup.
375 
376     -t_low T_low
377 
378         default: <the value from unit counts file>
379 
380         This parameter overrides the value contained in the unit
381         counts file. Since the unit counts file does not keep counts
382         of the units that appear fewer than T_low times in the
383         genome, it only makes sense to specify a T_low value that is 
384         greater than the one in the unit counts file. In that case 
385         unit with counts that are less than T_low will not be read 
386         from the unit counts file and the value of "-set_t_low" 
387         option will be used as the result of unit count lookup.
388 
389     -t_thres T_threshold
390 
391         default: <the value from unit counts file>
392 
393         All windows with score above T_threshold are masked. For the
394         definition of the window score see description of 
395         "-window_size" option. This parameter overrides the value
396         contained in the unit counts file.
397 
398     -text_match true/false
399 
400         default: true
401 
402         This option applies to "-exclude_ids" and "-ids" options.
403         If set to "false" the sequence ids are compared as instances
404         of CSeq_id classes. If set to "true" the sequence ids are
405         compared as strings. In that case each id is represented as
406         a sequence of words separated by '|' characters. A sequence
407         is found in "exclude_ids" or "ids" set if some element of
408         the set contains a subsequence of the given sequence which
409         spans that whole number of words. This option allows to 
410         overcome some problems resulting from direct CSeq_id 
411         comparisons.
412 
413     -window window_size
414 
415         default <use unit_size + 4>
416 
417         The WinMask module works by tracking the score of a sliding
418         window which is defined as average of counts of all units 
419         within the window. This parameter defines the size of the 
420         sliding window. By default the value unit_size + 4 is used so 
421         there are 5 units in a window.

423 FILES
424 
425     This section describes the file formats that are used by
426     WindowMasker and provides some examples (FASTA file format is not 
427     described here).
428 
429 
430     Genome in multiple FASTA files
431 
432     If the genome data is contained in multiple FASTA formatted
433     files, then "-fa_list true" should be used and the value of 
434     "-in" option should name the file containing the list of the 
435     required FASTA files, one file per line. For example, for human 
436     genome build 34, one could have one FASTA file per chromosome 
437     named chrX.fa, chrY.fa, chr1.fa, chr2.fa, ..., chr22.fa located 
438     in the /human34 directory. Then the file given to the "-in" 
439     option should look like this:
440 
441 /human34/chr1.fa
442 /human34/chr2.fa
443 ...
444 /human34/chr22.fa
445 /human34/chrX.fa
446 /human34/chrY.fa
447 
448     NOTE: Empty lines are ignored.
449 
450 
451     Unit Counts
452 
453     This file is the output of WindowMasker Stage 1 processing if
454     "ascii" unit counts format has been chosen.  The file is an ASCII
455     text file. It also serves as input for Stage 2 processing. When 
456     WindowMasker reads the unit counts file during Stage 2 empty lines 
457     and lines starting with '#' character are ignored. This way 
458     comments could be introduced into the unit counts file.
459 
460     The first line of the file contains one integer number which is 
461     the unit size.
462 
463     Then come the lines containing counts for the units which
464     appeared more than T_low times in the genome (and its reverse 
465     complement). These are ordered by the unit numerical value. Each 
466     line has the following structure:
467 
468                     UNIT_VALUE COUNT
469 
470     where UNIT_VALUE is an integer in hexadecimal format and COUNT is 
471     a decimal integer.
472 
473     The last section of the file contains the computed values for 
474     T_threshold, T_extend, T_low, and T_high, each on a separate line 
475     with the following structure:
476 
477                     >PARAM_NAME VALUE
478 
479     where PARAM_NAME is one of t_low, t_extend, t_threshold, t_high 
480     and VALUE is the integer value of the corresponding parameter.
481 
482     For example the unit counts file for human build 34 looks like 
483     this (only 20 lines in the beginning and 5 lines in the end are 
484     shown):
485 
486 
487 15
488 0 154
489 1 154
490 2 154
491 3 154
492 4 154
493 <...>
494 1a0 154
495 1a1 47
496 1a2 95
497 1a3 64
498 1a4 71
499 1a5 101
500 1a7 80
501 1a8 101
502 1a9 63
503 1aa 100
504 <...>
505 
506 >t_low       16
507 >t_extend    57
508 >t_threshold 86
509 >t_high      154
510 
511     Optimized Unit Counts
512 
513     This file is an output of WindowMasker Stage 1 processing when
514     "oascii" unit counts format has been chosen. The file is also
515     used as an input for the Stage 2 processing via "-ustat" command
516     line option. 
517     
518     The first line of the file contains the number
519     AAAA and serves as a file format identifier.
520 
521     The second line must contain a single integer between 1 and 16
522     which is the unit size.
523 
524     The third line contains 4 integer values separated by spaces:
525 
526        the number of units with collisions M (32 bit integer)
527        the hash key size (8 bit integer)
528        the right offset r of the hash key (8 bit integer)
529        the width w (in bits) of the field containing the number of 
530             collisions (8 bit integer)
531 
532     The next 4 lines contain windowmasker thresholds, one per
533     line in the following order:
534 
535     T_low
536     T_extend
537     T_threshold
538     T_high
539 
540     Then go 2^k 32-bit integers, one per line that for the hash
541     table. Each integer has the following form:
542 
543     The low order w bits contain the number c of units that have
544     a hash key equal to the index of this integer in the table.
545 
546         If c = 0 then the rest of the bits are 0.
547         If c = 1 then bits [w,23] contain the count of the uniq
548                  unit with this hash key. Bits [24-31] contain
549                  the bits [0,r-1] and [k + r,31] of the unit
550                  value.
551         If c > 1 then bits [w,31] contain the index of the start
552                  of the list of counts in the counts table (see
553                  below) of units with this hash key.
554 
555     The last M lines contain 1 16-bit integer each and form the counts
556     table. Each number has the following form:
557 
558         bits [0-8] - unit count;
559         bits [9-15] are bits [0,r-1] and [k + r,31] of the
560             corresponding unit value.
561 
562     Note that unit counts file in "oascii" format contains exactly
563     10 + 2^k + M lines.

578     WindowMasker Interval Output Format
579 
580     By default the output of WindowMasker Stage 2 is in Interval
581     format ("-outfmt interval"). The file is an ASCII text file
582     consisting of blocks of information for each input sequence in 
583     the order those sequences appear in the input FASTA file.  Each 
584     block starts with the FASTA title of the sequence followed by the 
585     description of masked intervals, one interval per line. The 
586     intervals do not overlap and are sorted by their start position.

589

651 SAMPLE WINDOWMASKER SESSION
652 
653     Assuming the list of genome files for human build 34 are in the 
654     file ./fa_list the following command will run the first stage of 
655     WindowMasker. Checking for duplicates is requested, so all 
656     candidates for duplicates will be reported on standard error 
657     (stderr). The possible duplicates reported below turn out to be 
658     false positives. During Stage 1, WindowMasker also shows the 
659     progress by printing dots to standard error (stderr).
660 
661 
662 ./windowmasker -mk_counts -fa_list true -in fa_list -checkdup true -out ustat.15
663 Possible duplication of sequences:
664 subject: lcl|Hs1_27103_34 and query: lcl|Hs1_33153_34
665 at intervals
666 subject: 5539 --- 35539
667 query  : 36748455 --- 36778455
668 
669 Possible duplication of sequences:
670 subject: lcl|Hs1_78005_34 and query: lcl|Hs1_78000_34
671 at intervals
672 subject: 173111 --- 233111
673 query  : 26192 --- 86192
674 
675 <...MANY MESSAGES SKIPPED...>
676 
677 Computing the genome length........................done.
678 Pass 1................................................................................................
679 Pass 2................................................................................................
680     
681 
682     At this point file ustat.15 contains the unit counts
683     that can be used in Stage 2.
684 
685     To mask the first human chromosome applying both WinMask
686     and symmetric DUST, the following command can be used assuming 
687     chr1.fa is in the current directory. During Stage 2 no 
688     progress information is printed on standard error (stderr).
689 
690 
691 ./windowmasker -ustat ustat.15 -in chr1.fa -out chr1.wm -dust true
692 
693 
694     The output in the interval format is written to chr1.wm.
